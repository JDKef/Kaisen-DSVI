# Plan de esquema PostgreSQL para Supabase

Estado: diseño para revisión. No se creó ninguna migración SQL ni se modificó
supabase/migrations.

## Decisiones de diseño

1. Supabase/PostgreSQL será la fuente de verdad de productos, stock y ventas.
2. Se conservarán las pantallas y los campos visibles actuales.
3. Los IDs de negocio serán numéricos: bigint para productos, ventas y líneas de
   venta. UUID se reserva para usuarios de Supabase y para la clave de
   idempotencia del cliente.
4. Los productos se archivan; no se borran físicamente.
5. Las escrituras de producto y venta se harán mediante funciones controladas,
   no mediante UPDATE de stock desde el cliente.
6. RLS se aplicará a todas las tablas expuestas. Ninguna clave service_role
   llegará a Flutter.
7. No se creará una tabla outbox ni una replicación bidireccional offline.

## Límite de negocio

Aunque el producto actual se comporta como una cuenta compartida, el esquema
incluye un límite explícito de negocio para que RLS no dependa de que todos los
usuarios autenticados puedan leer toda la base.

### businesses

- id uuid primary key.
- nombre text not null.
- creado_por uuid references auth.users.
- created_at timestamptz not null default now().

### business_members

- business_id uuid references businesses on delete cascade.
- user_id uuid references auth.users on delete cascade.
- role text not null, inicialmente admin u operator.
- created_at timestamptz not null default now().
- primary key (business_id, user_id).

La primera instalación tendrá un negocio configurado. El alta con el flujo
actual de usuario/contraseña debe asignar la cuenta a ese negocio mediante un
proceso controlado de registro. Una política de invitaciones puede sustituir
esa asignación más adelante sin añadir una pantalla en esta fase.

## Tablas de identidad

### profiles

- id uuid primary key references auth.users on delete cascade.
- username text not null.
- username_normalized text not null unique.
- created_at timestamptz not null default now().
- updated_at timestamptz not null default now().

No se guarda password_hash en public. La contraseña la gestiona Supabase Auth.
El nombre visible continúa siendo username.

## Tabla products

Columnas propuestas:

- id bigint generated by default as identity primary key.
- business_id uuid not null references businesses.
- nombre text not null.
- precio numeric(12,2) not null check (precio >= 0).
- stock integer not null default 0 check (stock >= 0).
- categoria text not null.
- codigo_barras text null.
- activo boolean not null default true.
- deleted_at timestamptz null.
- version bigint not null default 1.
- created_at timestamptz not null default now().
- updated_at timestamptz not null default now().
- created_by uuid not null references auth.users.
- updated_by uuid not null references auth.users.

Índices y restricciones:

- índice parcial único por business_id y lower(codigo_barras) cuando el código
  no es null y activo es true.
- índice por business_id, activo, nombre.
- índice por business_id, updated_at.
- checks para stock y precio no negativos.

version aumenta en cada actualización aceptada. updated_at y updated_by se
actualizan en el mismo cambio. La combinación de version y expected_version
evita que un dispositivo con caché vieja sobrescriba otro.

## Tablas de ventas

Se recomienda separar el encabezado de la venta de sus líneas. La vista de
historial volverá a presentar una fila por línea, que es la forma que entiende
el modelo Venta actual.

### sales

- id bigint generated by default as identity primary key.
- business_id uuid not null references businesses.
- client_operation_id uuid not null.
- payload_hash text not null.
- seller_id uuid not null references auth.users.
- occurred_at timestamptz not null.
- total numeric(12,2) not null check (total >= 0).
- status text not null default 'completed', limitado a completed o voided.
- created_at timestamptz not null default now().

Restricción única: (business_id, client_operation_id). La clave permite que un
retry de la misma operación devuelva la venta existente sin volver a descontar
stock.

### sale_items

- id bigint generated by default as identity primary key.
- business_id uuid not null references businesses.
- sale_id bigint not null references sales on delete restrict.
- product_id bigint not null references products on delete restrict.
- producto_nombre text not null.
- categoria text not null.
- cantidad integer not null check (cantidad > 0).
- precio_unitario numeric(12,2) not null check (precio_unitario >= 0).
- line_total numeric(12,2) not null check (line_total >= 0).
- created_at timestamptz not null default now().

Restricción única: (sale_id, product_id). El RPC debe rechazar o consolidar
duplicados antes de insertar; el carrito actual ya suma un mismo producto.
Los snapshots preservan el historial aunque el producto se archive o cambie de
nombre, categoría o precio.

### inventory_movements

Tabla de auditoría recomendada:

- id bigint generated by default as identity primary key.
- business_id uuid not null references businesses.
- product_id bigint not null references products on delete restrict.
- sale_id bigint null references sales on delete restrict.
- sale_item_id bigint null references sale_items on delete restrict.
- delta integer not null check (delta <> 0).
- reason text not null, inicialmente sale, manual_adjustment o migration.
- actor_id uuid not null references auth.users.
- created_at timestamptz not null default now().

Índice por producto y fecha. Para una venta, una constraint o validación del RPC
debe impedir dos movimientos sale para la misma sale_item_id. Esto facilita
auditoría y reconciliación sin convertir la tabla en una segunda autoridad del
stock: products.stock es el valor actual y los movimientos explican cómo llegó
allí.

## Vista compatible con historial

Crear una vista sale_history con una fila por sale_items:

- id = sale_items.id para que Venta.idRemoto conserve una identidad numérica
  por línea.
- sale_id = sales.id para trazabilidad de la operación completa.
- producto_id = sale_items.product_id.
- producto_nombre = sale_items.producto_nombre.
- categoria = sale_items.categoria.
- cantidad = sale_items.cantidad.
- precio_unitario = sale_items.precio_unitario.
- fecha = sales.occurred_at.
- total = sale_items.line_total.

La adaptación Flutter puede continuar llenando Venta con idRemoto igual al id de
la línea. sales.id se conserva en la respuesta del RPC y en la vista, aunque el
modelo actual no lo muestre. No se debe usar sales.id como ID de cada fila:
varias líneas de un mismo carrito compartirían el mismo valor y se rompería la
deduplicación del modelo actual.

## Funciones de escritura

### Productos

Las funciones públicas propuestas son:

- create_product(nombre, precio, stock, categoria, codigo_barras).
- update_product(id, expected_version, nombre, precio, stock, categoria,
  codigo_barras).
- archive_product(id, expected_version).

Cada función:

1. Verifica auth.uid() y la pertenencia al business_id.
2. Verifica rol operator o admin.
3. Rechaza valores inválidos o un código de barras duplicado.
4. Devuelve la fila completa con id, version y timestamps.
5. Actualiza version de forma monotónica.

update_product no debe aceptar una versión vieja. Debe devolver un error de
conflicto que permita al repository recargar el producto; nunca debe ganar
silenciosamente el último dispositivo que escribió.

archive_product pone activo = false y deleted_at = now(). No modifica las
líneas históricas. La consulta normal de catálogo filtra activo = true.

### create_sale: RPC transaccional e idempotente

Firma conceptual:

~~~text
create_sale(client_operation_id uuid, occurred_at timestamptz, items jsonb)
  returns sale_id bigint, item_ids bigint[], replayed boolean
~~~

La función se publicará como SECURITY DEFINER, fijará search_path seguro,
validará auth.uid() y tendrá EXECUTE únicamente para authenticated.

Algoritmo obligatorio:

1. Normalizar y validar business_id a partir del usuario autenticado.
2. Validar que client_operation_id no sea nulo.
3. Calcular un hash canónico del payload de producto_id y cantidad.
4. Buscar una venta existente por business_id y client_operation_id.
   - Si existe con el mismo hash, devolver sus IDs y replayed = true.
   - Si existe con otro hash, devolver conflicto de idempotencia.
5. Validar que items no esté vacío, que cada cantidad sea positiva y que no
   haya producto repetido.
6. Ordenar los product_id antes de bloquearlos con SELECT ... FOR UPDATE. El
   orden determinista reduce deadlocks entre dos carritos.
7. Verificar que todos los productos pertenezcan al negocio, estén activos y
   tengan stock suficiente.
8. Tomar precio, nombre y categoría desde products. El cliente no puede
   imponer precio ni snapshot.
9. Insertar sales.
10. Insertar sale_items con los snapshots y los importes calculados en
    numeric.
11. Actualizar cada products.stock = stock - cantidad y aumentar version.
12. Insertar inventory_movements con delta negativo y el vínculo a venta y
    línea.
13. Confirmar la transacción y devolver sale_id e item_ids.

Si falla cualquier validación o inserción, PostgreSQL hace rollback de todos los
cambios de la función. Dos dispositivos que venden el mismo último artículo
quedan serializados por el lock: uno termina y el otro recibe stock
insuficiente. Un timeout del cliente puede repetir exactamente el mismo RPC sin
crear otra venta.

El cliente debe generar client_operation_id antes de enviar el carrito y
conservarlo mientras exista una respuesta incierta. Retener una operación
transitoria para repetir la misma solicitud no es sincronización offline:
requiere red para ejecutar el RPC y no crea una cola de cambios locales. No se
permiten ventas nuevas cuando el dispositivo está desconectado.

## RLS y exposición

Todas las tablas públicas deben tener RLS habilitado. La función auxiliar
is_business_member(business_id) debe ser estable, segura frente a recursión de
políticas y usar un search_path explícito.

| Recurso | SELECT | INSERT/UPDATE/DELETE desde cliente |
|---|---|---|
| profiles | Solo el propio perfil; no expone hashes | No directo; registro controlado |
| businesses | Miembros del negocio | Solo bootstrap/admin |
| business_members | El propio membership; admin puede ver miembros de su negocio | Solo función administrativa |
| products | Miembros; catálogo normal solo activo | No directo; usar RPC |
| sales | Miembros del mismo negocio | No directo; solo create_sale |
| sale_items | Miembros del mismo negocio | No directo |
| inventory_movements | Miembros; opcionalmente solo admin para auditoría | No directo |

Las funciones de escritura deben comprobar pertenencia aunque sean
SECURITY DEFINER. Se deben revocar privilegios de ejecución al rol anon y
conceder únicamente lo necesario a authenticated. Las vistas de lectura deben
usar la semántica de seguridad invoker o quedar protegidas indirectamente por
las políticas de sus tablas.

## Autenticación compatible con la UI actual

La pantalla actual solo captura nombre de usuario y contraseña; no se debe
obligar a añadir email en esta fase. La estrategia propuesta es:

- Supabase Auth gestiona la contraseña, las sesiones y los refresh tokens.
- El repository normaliza username y deriva un alias interno determinista con
  formato de email en un dominio no entregable, por ejemplo
  u-<hash>@auth.kaisen.invalid.
- register_username, una Edge Function controlada, crea el usuario confirmado,
  profile y membership; si falla una parte, compensa la cuenta incompleta.
- Login usa el alias derivado con signInWithPassword.
- El alias nunca se muestra en la UI y service_role solo existe en la función.
- Como no hay email visible, recuperación de contraseña requiere una política
  operativa explícita; no se debe fingir que existe un email de recuperación.

La contraseña SHA-256 existente en SQLite no es un formato importable como
contraseña de Supabase Auth. La migración puede aprovechar la contraseña que el
usuario escribe en su primer login para crear la identidad remota; si no se
dispone de esa contraseña, se requiere registro o restablecimiento manual. No
se deben subir hashes locales a una tabla pública.

## Orden lógico de migraciones

Cuando el plan sea aprobado, las migraciones SQL deberían separarse así:

1. Extensiones, helper functions y tablas de identidad/negocio.
2. products, índices y triggers de versionado.
3. sales, sale_items e inventory_movements.
4. vistas de historial y funciones de lectura.
5. RLS, grants y funciones de productos.
6. create_sale y pruebas SQL de idempotencia/concurrencia.
7. Objetos privados de importación y compatibilidad, si se necesitan.

Estos son entregables futuros, no archivos que deban crearse en esta fase.
